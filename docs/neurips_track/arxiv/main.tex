\begin{abstract}
We present a benchmark-oriented extension of a ledger-first market infrastructure prototype for studying market-design and execution behavior under settlement and risk constraints. The environment combines seedable agent-based order flow, configurable immediate, speed-bump, and frequent-batch-auction matching regimes, double-entry style account state transitions, and deterministic safety checks for conservation and non-negativity. We expose reproducible benchmark tasks that compare latency, throughput, spread, price impact, queue-priority advantage, and latency-arbitrage profit across market designs. In the current implementation, all generated scenarios preserve settlement invariants, while mechanism changes induce clear latency and market-quality tradeoffs. The result is a benchmark draft for market-infrastructure research that is separate from the original systems-paper line and is intended as a foundation for stronger benchmark-style evaluation.
\end{abstract}

\section{Introduction}
Electronic markets are shaped jointly by matching rules, latency structure, and settlement semantics. Market-design work on frequent batch auctions argues that short batching windows can reduce some latency-driven distortions in continuous markets \cite{budish2015fba}. In practice, many evaluation artifacts study matching rules in isolation and do not model the settlement layer, replay behavior, or safety checks that an actual market stack must satisfy.

This paper develops a benchmark-oriented layer on top of an existing ledger-first market-infrastructure prototype. It is not a replacement for the original systems paper. It is a separate evaluation track that asks a narrower question: can market-design tradeoffs be measured inside an environment that preserves explicit settlement invariants?

The current paper makes four concrete contributions:
\begin{enumerate}[leftmargin=*]
  \item It defines a ledger-aware benchmark environment in which market-design experiments are coupled to deterministic settlement checks.
  \item It provides a seedable multi-agent order-flow generator with four agent classes, eleven benchmark scenarios, and explicit agent/workload sweeps.
  \item It reports single-seed and eight-seed aggregate benchmark results over throughput, latency, spread, price impact, queue-priority advantage, and arbitrage-profit proxies.
  \item It exposes a step-wise \texttt{Reset/Step/Observe/Metrics} API, a gym-style adapter with five runtime control channels, two adapter-driven policy baselines, and ablation suites for both market-structure toggles and agent/workload perturbations.
\end{enumerate}

\section{Problem Setting}
We study a discrete-time environment in which a market state is advanced one simulator step at a time. At step $t$, the state is represented as
\[
s_t = (B_t, S_t, X_t, f_t),
\]
where $B_t$ and $S_t$ denote the current buy and sell books, $X_t$ is the vector of account states, and $f_t$ is the latent fundamental used by the synthetic order-flow generator. A scenario additionally fixes a matching mode, a batch-window size, risk thresholds, and a population of agents.

The benchmark focuses on the following research questions:
\begin{enumerate}[leftmargin=*]
  \item How do immediate, speed-bump, adaptive, and frequent-batch-auction execution differ in latency and fill behavior?
  \item Can fairness-adjacent signals such as queue-priority advantage and latency-arbitrage profit be measured reproducibly under deterministic settlement semantics?
  \item Do settlement invariants hold under heterogeneous agents and stress scenarios?
\end{enumerate}

\section{Benchmark Environment}
\subsection{Core Components}
The benchmark environment lives in the repository's \texttt{simulator/} package. It consists of scenario, order, fill, and result schemas, heterogeneous agent order generation, immediate, speed-bump, adaptive, and batch-clearing matching paths, deterministic settlement application, and metric accumulation.

The current environment also exposes a step-wise control surface with \texttt{Reset()}, \texttt{Observe()}, \texttt{Step()}, and \texttt{Metrics()}. On top of that surface, the repository includes an adapter layer that returns reward-bearing timesteps and advertises per-scenario action support. In the current implementation, the adapter exposes five explicit control channels: batch-window override for adaptive scenarios, risk-limit scaling, tie-break randomization, release-cadence control, and price-aggression bias.

\subsection{Determinism and Invariants}
Each scenario is seeded. Given the same scenario configuration and seed, the environment is expected to produce the same accepted order counts, fill counts, and proxy metrics.

The benchmark currently enforces three operational invariants:
\begin{enumerate}[leftmargin=*]
  \item \textbf{Conservation}: total cash and total inventory across accounts remain constant over time.
  \item \textbf{Non-negativity}: no account is allowed to end a fill application with negative cash or negative units.
  \item \textbf{Deterministic replay}: seeded runs produce stable outputs for the same configuration.
\end{enumerate}

\section{Agent Models}
The present environment includes four agent classes:
\begin{itemize}[leftmargin=*]
  \item \textbf{Market makers}: quote on both sides around the current fundamental with configurable width and intensity.
  \item \textbf{Latency arbitrageurs}: react to stale price information and attempt to exploit short-horizon dislocations.
  \item \textbf{Retail traders}: submit noisier buy or sell orders with wider dispersion around the current reference level.
  \item \textbf{Informed traders}: trade in the direction of the next-step fundamental when the generator exposes a directional signal.
\end{itemize}

These agents are intentionally simple. Their purpose is to generate heterogeneous but controlled activity so that matching rules can be compared under a repeatable workload.

\section{Tasks and Baselines}
The current benchmark suite contains eleven scenarios:
\begin{enumerate}[leftmargin=*]
  \item \texttt{Immediate-Surrogate}
  \item \texttt{SpeedBump-50ms}
  \item \texttt{FBA-100ms}
  \item \texttt{FBA-250ms}
  \item \texttt{FBA-500ms}
  \item \texttt{Adaptive-100-250ms}
  \item \texttt{Adaptive-OrderFlow-100-250ms}
  \item \texttt{Adaptive-QueueLoad-100-250ms}
  \item \texttt{Policy-BurstAware-100-250ms}
  \item \texttt{Policy-LearnedBandit-100-250ms}
  \item \texttt{FBA-250ms-Stress}
\end{enumerate}

The benchmark now includes three heuristic adaptive-window baselines and two adapter-driven policy controllers. The burst-aware controller is hand-written. The learned baseline is a contextual bandit over discrete action bundles built from the expanded action space.

\section{Metrics}
The environment reports four groups of signals:
\begin{itemize}[leftmargin=*]
  \item \textbf{Systems metrics}: orders per second, fills per second, and p50/p95/p99 latency.
  \item \textbf{Market-quality metrics}: average spread and average price impact.
  \item \textbf{Fairness-adjacent proxies}: queue-priority advantage, latency-arbitrage profit, and execution dispersion.
  \item \textbf{Safety metrics}: negative-balance violations, conservation breaches, and risk rejections.
\end{itemize}

\section{Experimental Setup}
We report two result layers. The first is a single-seed reproducibility snapshot. The second is an eight-seed aggregate profile that is more appropriate for paper-facing discussion. The aggregate profile uses seeds
\[
[7, 11, 19, 23, 29, 31, 37, 41].
\]

All reported scenarios use the same simulator implementation and differ only in matching mode, batch-window size, seed, and population intensity. The current setup uses a discrete-time step duration of $10$ milliseconds.

\subsection{Observed Tradeoffs}
Three measured observations are currently the most informative.

First, immediate execution retains the lowest latency profile, with mean p50, p95, and p99 all equal to $10$ milliseconds under the present discrete-time environment.

Second, the $50$ millisecond speed-bump baseline introduces a fixed $60$ millisecond latency profile and lowers throughput to $1294.30$ orders per second while preserving the same queue-priority-advantage proxy observed under immediate execution. This makes it a useful intermediate baseline: latency rises, but batching-style fairness proxies do not automatically improve.

Third, the $250$ millisecond batch regime lowers the queue-priority-advantage proxy to $0.0273$, below immediate execution ($0.0742$), the speed-bump baseline ($0.0742$), and the $100$ millisecond batch regime ($0.0571$). The balanced adaptive heuristic settles around a mean window of $207.14$ milliseconds and lowers the arbitrage-profit proxy to $522.00$, below both fixed $100$ and fixed $250$ millisecond batching.

Fourth, the controller comparison now surfaces an internal benchmark tradeoff. The burst-aware controller saturates at a $250$ millisecond mean window and lands at lower queue-priority advantage ($0.0305$) and lower arbitrage-profit proxy ($621.00$). The learned bandit collapses to a fast $100$ millisecond control profile, improving fills to $743.06$ per second and reducing p99 to $155.00$ milliseconds, but giving back fairness-adjacent proxy scores ($0.0423$, $896.50$). This is exactly the sort of result the benchmark should expose: a controller can optimize latency and fill rate without improving fairness-adjacent outcomes.

The stress scenario increases throughput to $1769.25$ orders per second and $900.50$ fills per second. It also increases the arbitrage-profit proxy to $2057.00$, which shows that higher activity does not automatically imply improved fairness-adjacent outcomes.

Across all $11 \times 8 = 88$ aggregate runs, the environment reports zero negative-balance violations and zero conservation breaches.

\subsection{Ablation Snapshots}
We generate a targeted market-structure ablation suite over seeds $[13, 17, 19, 23]$ for a stressed constrained $250$ millisecond batch regime. Relaxing risk limits increases throughput from $1190.48$ to $1770.24$ orders per second and removes all risk rejections, but it also increases the arbitrage-profit proxy from $685.25$ to $2198.50$. Randomizing tie-breaks leaves throughput roughly unchanged while worsening p99 latency from $320.00$ to $402.50$ milliseconds. Disabling settlement checks does not change throughput in this setup, indicating that invariant enforcement is not the dominant bottleneck for the present benchmark scale.

We also generate an agent/workload sweep over seeds $[43, 47, 53, 59]$. Removing arbitrageurs collapses arbitrage-profit proxy to zero and flips queue advantage negative. Doubling arbitrage intensity pushes arbitrage-profit proxy to $1730.75$. Widening market-maker quotes pushes p99 to $485.00$ milliseconds and lowers fills to $595.63$. Increasing informed intensity raises throughput modestly while leaving arbitrage-profit much closer to control than the arbitrage sweep. These results show that the benchmark is sensitive to population composition and workload intensity, not only to the mechanism choice.

\section{Limitations}
The current artifact is still short of a strong top-tier benchmark submission. The most important limitations are:
\begin{itemize}[leftmargin=*]
  \item proxy fairness metrics rather than richer behavioral or welfare metrics
  \item the adapter action space is still narrow compared with a full exchange-control environment
  \item the learned controller is still a lightweight contextual bandit, not a richer policy-learning result
  \item the current agent/workload sweeps are still small and do not yet cover full parameter grids over market-maker or informed-flow behaviors
\end{itemize}

\section{Related Work}
The benchmark is motivated directly by frequent-batch-auction market design \cite{budish2015fba}. Its benchmark-oriented framing is also aligned with the broader machine-learning practice of exposing reusable environments rather than only single-task scripts \cite{brockman2016gym}. The present contribution sits between those two traditions: it is neither a pure market-design theory paper nor a standard reinforcement-learning benchmark, but an infrastructure-aware environment that aims to make later learning or control work more credible.

\section{Conclusion}
This NeurIPS-track benchmark line establishes a reproducible, ledger-aware evaluation environment for market-design experiments without overwriting the original systems-paper line. The current results already show measurable latency and market-quality tradeoffs across immediate, speed-bump, adaptive, adapter-driven policy, and batch matching regimes, while preserving executable settlement invariants. The next stage is clear: replace the contextual bandit with a stronger learned controller, broaden the action surface further, and introduce deeper sweep studies so that the environment can support more serious benchmark or learning-oriented submissions.
